# 图片上传与服装分析功能需求文档

## 1. 需求本质（第一性原理）

用户的核心目标是：**用一张衣服照片作为输入，让系统理解衣服是什么、有什么特征，然后自动生成对应的生产研发方案。**

拆解到最基础的能力：

1. **输入捕获**：用户需要把本地照片传到系统里。
2. **内容理解**：系统需要"看懂"图片，识别出服装的关键属性。
3. **结构化**：把识别结果转换成系统能理解的研发参数。
4. **方案生成**：用这些参数调用已有流程，产出面料、款式、成本、Tech Pack 等结果。

不需要重新发明推荐引擎、成本计算、BOM 生成，这些已经存在。新功能只解决"从图片到需求"这一步。

## 2. 用户场景

- 用户看到一件喜欢的衣服（竞品、样衣、街拍、电商图），拍下来或下载图片上传。
- 系统分析后告诉用户这是"女款春秋城市轻户外冲锋衣"，并生成完整研发方案。
- 用户可以基于此方案继续调整文本需求、导出 Tech Pack、保存历史。

## 3. 成功标准

- 用户能在生成器页面上传 1 张图片。
- 上传后系统返回分析出的服装信息（品类、性别、季节、场景、功能优先级、款式关键词、风格定位等）。
- 分析结果自动填入需求表单或作为文本补充，生成方案。
- 支持常见格式：jpg、jpeg、png、webp。
- 单张图片大小不超过 5MB。
- 如果图片分析失败，系统给出明确提示，并允许用户手动输入需求继续生成。

## 4. 技术方案

### 4.1 图片上传

- 前端使用 `<input type="file" accept="image/*">` 选择图片。
- 选中后先在本地预览，用户确认后再上传。
- 通过 `FormData` POST 到 `/api/analyze-image`。
- 后端使用 Next.js Route Handler 接收 multipart/form-data。
- 图片暂存在内存中（Buffer），不做持久化存储。

### 4.2 图片分析

- 使用 DashScope（通义千问视觉模型）作为图像理解引擎，与现有 AI 配置保持一致。
- 上传的图片转成 base64，调用视觉模型。
- 要求模型只返回 JSON，字段包括：
  - `category`：品类
  - `gender`：性别
  - `ageRange`：年龄段
  - `season`：季节
  - `scenes`：使用场景数组
  - `functionPriorities`：功能优先级数组
  - `stylePositioning`：风格定位
  - `styleKeywords`：款式关键词数组
  - `targetPrice`：目标价格（若图片无法判断，使用默认值）
  - `description`：对图片的简短描述
- 后端对 AI 返回结果做校验和归一化，确保字段符合 `ParsedRequirement` 约束。

### 4.3 与生成流程集成

- 分析成功后，前端把结果合并到现有需求表单中。
- 用户可以选择直接生成方案，或继续修改需求后再生成。
- 后端 `/api/generate` 无需修改，继续接收 `GenerateRequest`。

### 4.4 失败处理

- 文件过大：前端校验，超过 5MB 提示用户压缩。
- 非图片格式：前端校验。
- AI 分析失败：返回错误信息，前端保留图片预览并允许用户手动输入。
- AI 返回不完整：后端用合理默认值补齐，并在 constraints 中标注"图片未明确，系统默认"。

## 5. 界面设计

- 在需求输入区域增加"上传图片"按钮。
- 上传后显示图片缩略图和"分析中"状态。
- 分析完成后显示识别出的关键信息卡片。
- 提供"重新上传"和"清除图片"按钮。
- 如果用户同时输入了文字和图片，两者合并：文字优先的字段以文字为准，图片补充文字未提及的字段。

## 6. 数据安全

- 图片仅在本次请求中使用，不保存到服务器磁盘或数据库。
- 调用 DashScope 时遵循现有环境变量配置，API key 不泄露到前端。

## 7. 不在这个范围内的

- 不上传多张图片。
- 不做服装轮廓分割或关键点检测（只做多模态语义识别）。
- 不把图片保存到历史记录中。
- 不新增独立的图片模型训练流程。
