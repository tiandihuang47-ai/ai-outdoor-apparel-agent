# AI 户外服饰智能设计 Agent V2 设计文档

## 1. 项目背景与目标

当前版本已实现核心 AI 方案生成、效果图生成、导出分享、历史记录、CI/CD 与自动部署。V2 目标是在保留现有能力的基础上，提升产品完整度与商业化体验：

- 提升视觉专业度与操作引导
- 扩展可设计服装品类与场景
- 增加用户身份与跨设备数据同步

## 2. 范围

### 2.1 本次做

1. **UI/UX 全面升级**：深色科技风 + 可视化流程 + 玻璃拟态卡片
2. **服装品类模板扩展**：
   - 补充热门缺失品类
   - 现有品类按基础/中端/高端细分
   - 增加场景化套餐模板
3. **用户登录与云端历史记录**：
   - Vercel KV 云端存储
   - 邮箱验证码登录
   - 游客模式与登录后合并

### 2.2 本次不做

- 接入新的 AI 文本/图像模型
- 第三方登录（Google/GitHub 等）
- 支付、会员体系、团队协作
- 后端数据库（仍用 KV 存储）

## 3. 阶段实施计划

### 阶段 1：UI/UX 全面升级

#### 3.1.1 设计系统

- **主色调**：深 slate 背景（#0f172a / #1e293b），强调色为蓝紫渐变（#6366f1 → #06b6d4）
- **卡片**：玻璃拟态效果（半透明背景 + 模糊 + 细边框）
- **字体**：系统字体栈，标题加粗，正文 14-16px，辅助文字 12-13px
- **圆角**：卡片 16px，按钮 12px，标签 999px
- **阴影/光效**：微妙的内发光、边框微光、hover 时卡片上浮

#### 3.1.2 全局组件

- `GlassCard`：玻璃拟态卡片容器
- `StepWizard`：步骤向导（输入需求 → 生成方案 → 查看结果 → 导出/分享）
- `SkeletonLoader`：骨架屏加载
- `AnimatedButton`：带加载状态与成功反馈的按钮
- `Toast`：全局操作反馈
- `EmptyState`：空状态插画 + 提示

#### 3.1.3 页面改造

- **首页 `/`**：
  - 顶部 Hero 区域 + 核心功能入口
  - 步骤向导悬浮或置顶
  - 需求表单使用玻璃卡片包裹
  - 生成结果分区展示（方案、面料、成本、营销、风险）
- **生成器页 `/generator`**：
  - 三栏布局：需求表单 | 方案结果 | 历史/对比
  - 每个结果模块可折叠/展开
- **历史面板**：
  - 卡片式列表
  - 支持收藏、批量选择、搜索/筛选
- **设置弹窗**：
  - 保持现有功能，视觉上统一为玻璃卡片风格

#### 3.1.4 交互优化

- 生成方案时显示骨架屏 + 步骤提示
- 生图按钮显示进度条或倒计时
- 导出/分享操作显示成功/失败 Toast
- 表单校验错误聚焦并显示具体提示

### 阶段 2：服装品类模板扩展

#### 3.2.1 补充热门品类

新增以下品类模板（每个品类至少 2-3 款）：

- 棒球服 / 校队夹克
- 家居服 / 睡袍
- 户外背包（作为配饰品类试点）
- 户外鞋（徒步鞋、越野跑鞋）
- 腰带 / 围巾 / 手套 / 帽子等配饰补强

#### 3.2.2 价格档次细分

为每个核心品类增加三档模板：

- **基础款**：功能最简、成本最低、走量定位
- **中端款**：平衡功能与成本，主流价格带
- **高端款**：专业功能、高端面料、高溢价

#### 3.2.3 场景化套餐模板

新增套餐类型，一次生成多套搭配：

- 露营全套穿搭（冲锋衣 + 软壳 + 速干T恤 + 徒步裤 + 配件）
- 城市通勤一周穿搭（5 套日常搭配）
- 冬季保暖套装（羽绒服 + 保暖内衣 + 围巾手套）
- 夏季防晒户外套装（防晒衣 + 速干T恤 + 短裤 + 帽子）

#### 3.2.4 模板数据结构

扩展现有 `StyleTemplate`，新增字段：

- `tier: 'basic' | 'mid' | 'premium'`
- `isPackage: boolean`
- `packageItems?: string[]`（套餐包含的子品类 ID）
- `sceneTags: string[]`

#### 3.2.5 UI 适配

- 需求表单增加"品类"和"场景套餐"下拉选择
- 结果页支持套餐类型展示（多子方案折叠/展开）
- 历史记录区分单方案与套餐

### 阶段 3：用户登录与云端历史记录

#### 3.3.1 技术选型

- **存储**：Vercel KV（Redis 兼容，免费额度足够个人/小团队使用）
- **认证**：自研邮箱验证码登录（无需依赖第三方 OAuth）
- **会话**：Next.js API Route + HTTP-only Cookie
- **数据隔离**：每个用户独立 namespace

#### 3.3.2 数据模型

```
KV key: user:<email>:profile       -> { email, createdAt, lastLoginAt }
KV key: user:<email>:history       -> HistoryItem[]
KV key: user:<email>:favorites     -> string[]
KV key: verif:<email>:<code>       -> { expiresAt }  // 验证码临时存储
```

#### 3.3.3 API 设计

- `POST /api/auth/send-code`：发送邮箱验证码
- `POST /api/auth/verify-code`：验证验证码并设置登录 Cookie
- `POST /api/auth/logout`：清除登录态
- `GET /api/auth/me`：获取当前登录用户信息
- `GET /api/history`：获取云端历史记录
- `POST /api/history`：保存/覆盖云端历史记录
- `POST /api/history/sync`：合并本地与云端历史记录

#### 3.3.4 前端改造

- 新增 `AuthModal` 登录弹窗（邮箱输入 → 验证码输入 → 登录成功）
- 顶部导航显示登录状态与头像/邮箱
- 历史记录优先读取云端，未登录时回退到 localStorage
- 登录时提示"是否合并本地记录到云端"

#### 3.3.5 安全与隐私

- 验证码 6 位数字，5 分钟有效
- 同一邮箱 1 分钟内不能重复发送
- Cookie 设置 `HttpOnly`、`Secure`、`SameSite=Lax`
- KV 中不存储用户密码，降低泄露风险

┌─────────────────────────────────────────────┐
│                 Next.js App                  │

## 5. 测试计划

- 每个新工具函数/Hook 补充单元测试
- 登录流程需测试：发送验证码、验证过期、重复发送限制
- 历史同步需测试：空云端、空本地、冲突合并
- UI 组件用 Vitest + jsdom 做基础渲染测试
- 每次提交前运行 `npm test`（已有 pre-commit hook）

## 6. 部署与发布

- 每个阶段完成后 push 到 main，Vercel 自动部署
- 阶段 3 需在 Vercel 后台配置 `KV_URL`、`KV_REST_API_URL`、`KV_REST_API_TOKEN`
- 邮箱验证码发送需配置 SMTP 或邮件服务（如 Resend/ SendGrid）

## 7. 风险与回退

- Vercel KV 免费额度用尽：可降级为 localStorage + 导出备份
- 邮件服务不可用：暂时关闭登录功能，保持游客模式
- UI 改造影响现有功能：分组件逐步替换，保留原页面作为回退

## 8. 验收标准

- [ ] 首页与生成器页视觉上统一为玻璃拟态风格
- [ ] 生成方案时有明确的步骤引导和加载反馈
- [ ] 模板总数超过 120 个，覆盖套餐类型
- [ ] 用户可通过邮箱验证码登录并同步历史记录
- [ ] 所有测试通过，CI/CD 正常
